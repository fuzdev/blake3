# blake3

> BLAKE3 hashing compiled to WASM from the `blake3` Rust crate

Own WASM build of BLAKE3, packaged as `@fuzdev/blake3_wasm` and `@fuzdev/blake3_wasm_small`. Uses
`-O3` optimization with the blake3 crate's hand-optimized WASM SIMD implementation (`wasm32_simd`
feature) for best throughput, plus a size-optimized build without SIMD.

**Repository**: `github.com/fuzdev/blake3`

## Quick Start

```bash
deno task check                    # Rust: typecheck + test:rust + clippy + cargo fmt --all + deno fmt
deno task build:wasm               # Build all WASM targets
deno task compare                  # Build + correctness: all builds vs test vectors
deno task test                     # All correctness tests (requires built artifacts)
deno task validate                 # Full validation: check + build + test + validate:*
deno task bench                    # Full benchmarks: Deno + Node.js + Bun + Wasmtime + report
```

## Crates

### blake3_wasm_core

Shared rlib containing the wasm-bindgen exports and shared TS (`types.ts`, `stream.ts`).
Both `blake3_wasm` and `blake3_wasm_small` depend on this crate via `pub use blake3_wasm_core::*;`.
Has a `simd` feature that enables `blake3/wasm32_simd`. See `crates/CLAUDE.md` for details.

**One-shot:** `hash`, `keyed_hash`, `derive_key`

**Streaming:** `Blake3Hasher` with `new`, `new_keyed`, `new_derive_key`, `update`, `finalize`, `finalize_and_reset`, `reset`, `free`, `[Symbol.dispose]`

**Async streams:** `hash_stream`, `keyed_hash_stream`, `derive_key_stream` — convenience functions that hash a `ReadableStream<Uint8Array>` via the streaming hasher. Defined in `stream.ts` and re-exported from all entry points.

**Limitation:** All outputs are fixed at 32 bytes (256 bits). BLAKE3 supports variable-length output (XOF) but this API does not currently expose it.

### blake3_wasm

WASM bindings — SIMD build. Depends on `blake3_wasm_core` with `simd` feature. Compiled with `-O3`
and blake3's `wasm32_simd` for hand-optimized WASM SIMD compression.

Supports TC39 explicit resource management:

```ts
using hasher = new Blake3Hasher();
hasher.update(data);
const digest = hasher.finalize();
// hasher.free() called automatically
```

Stream example:

```ts
const digest = await hash_stream(file.stream());
```

### blake3_wasm_small

Size-optimized WASM bindings — identical API to blake3_wasm but depends on `blake3_wasm_core`
without the `simd` feature, compiled with `-Os` optimization. Produces a ~32 KB binary (vs ~47 KB
for the SIMD build). Serves Bun users (where WASM SIMD is slower) and bundle-size-sensitive contexts.

**Tradeoffs vs blake3_wasm:** 30% smaller binary, ~2.6x slower on Deno/Node at 64KB+ inputs,
similar speed at small inputs, 1.6-2.6x _faster_ on Bun (due to Bun's WASM SIMD regression; worst at 512B–3KB).

### blake3_component

WASI component implementing the `fuzdev:blake3` WIT interface. Zero imports — pure computation only.

**WIT interface** (`wit/blake3.wit`): Defines `hashing` interface with `hash-error` enum, one-shot functions (`hash`, `keyed-hash` returning `result`, `derive-key`), and a `hasher` resource for streaming (`new-keyed` returns `result`).

**Build**: `cargo component build -p blake3_component --release` (requires `cargo-component`)

### blake3_bench_wasmtime

Benchmarks and correctness testing for BLAKE3 WASI components using the Wasmtime runtime with typed component model bindings.

**Benchmark mode** (default): Benchmarks hash, keyed_hash, derive_key, and streaming. Writes results in the same `BenchSuiteResult` JSON format as the TS benchmarks, plus per-run markdown history files.

**Compare mode** (`--compare`): Verifies all component operations (hash, keyed_hash, derive_key, streaming, keyed streaming, derive_key streaming) against test vectors. Also tests error paths (invalid key length returns `Err`, not panic).

The web target serves both npm publishing and Node.js usage. `mod_node.ts` imports `initSync` from

target needed. The npm wrapper files re-export as `init_sync` for snake_case consistency.

### `deno compile` Compatibility

wasm-bindgen's deno target uses `fetch()` to load WASM, which is incompatible with `deno compile` (embedded files are only accessible via `Deno.readFile*`, and `--include` on `.wasm` files triggers module resolution on the WASM's import section). The `build:wasm:deno` and `build:wasm:small:deno` tasks automatically run `scripts/patch_deno_compile.ts` after `wasm-pack` to:

1. Replace `fetch()` + `WebAssembly.instantiateStreaming` with `Deno.readFileSync` + `WebAssembly.instantiate`
2. Create a `_bg.js` stub to satisfy compile-time WASM module import resolution

Consumers include the `pkg/deno/` directory via `--include` in their `deno compile` flags so the WASM binary is embedded.

## Benchmarks

See `benches/CLAUDE.md` for detailed benchmark docs, WASM boundary crossing analysis,
and improvement opportunities.

The `npm:blake3-wasm` package (v2.1.5) is installed as a dev dependency for comparison.

**Correctness** (`deno task test:deno`): Verifies blake3_wasm output (hash, keyed_hash, derive_key, streaming, stream convenience functions, stream batch boundaries) against native blake3 test vectors, including error path tests for invalid key length (`keyed_hash`, `new_keyed`, `keyed_hash_stream`). Uses `Deno.test` for structured test output and `--filter` support. `deno task test:deno:small` runs the same tests against blake3_wasm_small. `deno task test:component` does the same for the WASI component via Wasmtime. `deno task test` runs all correctness tests.

**Performance** (`deno task bench`): Full pipeline — builds WASM, runs benchmarks in Deno, Node.js, Bun, and Wasmtime, produces a cross-runtime comparison report.

### Runners

JS runtimes (Deno, Node.js, Bun) benchmark 3 implementations across hash, keyed_hash, derive_key (each at 4 sizes), streaming (3 sizes: 1 KB/64 B chunks, 64 KB/8 KB chunks, 1 MB/8 KB chunks), and stream convenience functions (hash_stream, keyed_hash_stream, derive_key_stream at the same 3 streaming sizes):

1. `blake3_wasm` — our SIMD build (category: `blake3`)
2. `blake3_wasm_small` — our size-optimized build, no SIMD (category: `blake3`)
3. `npm:blake3-wasm` — reference package (category: `reference`)

Wasmtime benchmarks our component via the WASI component model, covering hash, keyed_hash, derive_key, and streaming (same 3 sizes).

Each runner has a `category` field (`blake3`, `reference`) persisted in bench results for structured report generation.

**Comparison caveats:**

- The npm:blake3-wasm `keyedHash`/`deriveKey` use the streaming API internally (create hasher → update → digest = 3 wasm boundary crossings), while our `keyed_hash`/`derive_key` are one-shot wasm exports (1 call). This API design difference explains the large gap at small input sizes — it's real-world performance, not a raw algorithm difference.
- Deno has ~5-9x higher per-call WASM overhead at small inputs compared to Node.js (up to ~8x for hash at 32B). The wasm-bindgen glue code is identical between targets — this is a Deno runtime characteristic. The gap narrows to ~1x at 1MB.
- Bun has a WASM SIMD regression — blake3_wasm (SIMD) is 1.6-2.6x slower than blake3_wasm_small (no SIMD) on Bun (worst range: 512B–3KB, ~1.44x at 64KB+), while SIMD is ~2.6x faster on Deno/Node at 64KB+. No upstream issue filed yet.

### Architecture

Shared engine in `benches/lib/bench_core.ts` (`make_runners`, `run_benchmarks`). Entry points provide runtime-specific imports:

- `benches/deno/bench.ts` — Deno (imports from mod.ts)
- `benches/node/bench.ts` — Node.js/Bun (imports from mod_node.ts, `BENCH_RUNTIME` env var)
- `crates/blake3_bench_wasmtime/src/main.rs` — Wasmtime (typed bindings via `wasmtime::component::bindgen!`)
- `benches/report.ts` — auto-discovers `benches/results/*.json`, produces cross-runtime comparison with throughput and ±CI. Every run saves timestamped history files alongside fixed-name outputs
- `benches/diff.ts` — regression detection between two result files using Welch's t-test and Cohen's d. Auto-detects latest two timestamped files or accepts explicit paths

## WIT Interface

The `fuzdev:blake3` WIT package (`wit/blake3.wit`) defines the component model interface:

- **World `blake3`**: exports the `hashing` interface, imports nothing (pure computation)
- **Interface `hashing`**: `hash-error` enum, one-shot functions (`keyed-hash` returns `result<list<u8>, hash-error>`), `hasher` resource for streaming (`new-keyed` returns `result<hasher, hash-error>`)
- Built with `cargo-component`, consumed by `wasmtime::component::bindgen!`

## Tooling Requirements

- **`cargo-component`**: `cargo install cargo-component` — builds WASM components from Rust
- **`wasm32-wasip1` target**: `rustup target add wasm32-wasip1`
- **`wasm-pack`**: for wasm-bindgen builds (JS runtimes)

## Conventions

- **Rust edition 2024**, resolver 2
- **Strict clippy**: all + pedantic + nursery + cargo enabled
- **`unsafe_code = "forbid"`** at workspace level
- **Release profile**: workspace uses opt-level=s, WASM builds override to `-O3` via RUSTFLAGS
- **WASM SIMD**: blake3_wasm depends on blake3_wasm_core with `simd` feature + `-C target-feature=+simd128` in RUSTFLAGS
- **WASM small**: blake3_wasm_small uses `-C opt-level=s` (no SIMD) with wasm-opt `-Os`
- **wasm-opt**: blake3_wasm uses `-O3` with `--enable-simd`, blake3_wasm_small uses `-Os` (no SIMD flags)
- **Deno formatting**: tabs, 100 line width, single quotes
- **Cross-runtime TS**: use `node:process`, `node:fs` (Deno supports these)
- **RUSTFLAGS** for WASM optimization (wasm-pack doesn't support `--profile`)
- **WIT-first design**: component interfaces defined in WIT, implemented via `wit-bindgen`
- **No backwards compatibility** pre-1.0 — breaking changes are fine
- **No new dependencies** unless explicitly requested
- User manages all git operations

## Optimization History

### Full optimization stack

| Layer          | Setting                                                                                                                                 | Effect                                                                |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| Cargo profile  | `opt-level = "s"`                                                                                                                       | Size-optimized (overridden by RUSTFLAGS for WASM)                     |
| Cargo profile  | `lto = true`                                                                                                                            | Full link-time optimization, dead code elimination                    |
| Cargo profile  | `codegen-units = 1`                                                                                                                     | Better cross-crate optimization                                       |
| Cargo profile  | `panic = "abort"`                                                                                                                       | No unwinding machinery                                                |
| Cargo profile  | `strip = true`                                                                                                                          | No debug symbols                                                      |
| RUSTFLAGS      | `-C opt-level=3`                                                                                                                        | Speed-optimized (overrides profile for WASM builds)                   |
| RUSTFLAGS      | `-C target-feature=+simd128`                                                                                                            | Enables WASM SIMD                                                     |
| wasm-opt       | `-O3 --enable-simd --enable-bulk-memory --enable-nontrapping-float-to-int --enable-mutable-globals --enable-sign-ext --strip-producers` | Speed-optimized post-processing, enable WASM features, strip metadata |
| blake3 feature | `wasm32_simd`                                                                                                                           | Hand-written SIMD compression                                         |
| blake3 dep     | `default-features = false`                                                                                                              | Omit `std` feature (not needed for WASM cdylib)                       |

### SIMD: the critical optimization

The build uses `-O3` with blake3's `wasm32_simd` feature for hand-optimized WASM SIMD compression.
Without the `wasm32_simd` feature, both `-Os` and `-O3` builds used blake3's portable Rust code
and performed ~2.5x slower at medium/large inputs. The SIMD feature was the critical optimization.

With SIMD enabled, we benchmarked `-Os` vs `-O3`. The differences were small (5-17%) but `-O3`
consistently won at small inputs (the most common hot path) with only ~5% binary size cost.
Consolidated to one `-O3` package.

### Binary sizes (measured)

| Build                              |    Size | vs npm:blake3-wasm |
| ---------------------------------- | ------: | -----------------: |
| blake3_wasm (SIMD, `-O3`)          | 46.8 KB |      +13,532 bytes |
| blake3_wasm_small (no SIMD, `-Os`) | 31.8 KB |       -1,827 bytes |
| npm:blake3-wasm                    | 33.6 KB |           baseline |

The ~15 KB size difference between blake3_wasm and blake3_wasm_small is the SIMD code:

- **SIMD code**: ~10-12 KB — blake3's hand-written `wasm32_simd.rs` compression routines
- **`-O3` vs `-Os`**: ~2-3 KB — aggressive inlining/unrolling for speed
- **Miscellaneous**: ~1 KB

blake3_wasm_small is 1,827 bytes smaller than the npm reference — effectively the same binary size.

Additional size optimizations applied:

- `--strip-producers` in wasm-opt: removes compiler metadata section (~26 bytes)
- `default-features = false` for blake3: omits `std` feature (no measurable WASM size change,
  but cleaner dependency — `blake3_debug` native binary adds `std` back explicitly)

### Bun WASM SIMD regression (confirmed)

Bun's WASM engine has poor SIMD performance — confirmed by blake3_wasm_small benchmarks:

- **Bun**: blake3_wasm (SIMD) is 1.6-2.6x _slower_ than blake3_wasm_small (no SIMD). The regression is
  worst in the 512B–3KB range (~2.3–2.6x penalty), moderate at 32B (~1.6x) and 64KB+
  (~1.44x). The 4KB boundary marks a notable improvement — BLAKE3's 4-chunk SIMD parallelism
  partially offsets the overhead (see Q7 in benches/CLAUDE.md).
- **Deno/Node.js**: blake3_wasm (SIMD) is ~2.6x _faster_ than blake3_wasm_small at 64KB+ inputs.
  At small inputs (32B, 1KB) the difference is ~1x on Deno; ~1.1-1.2x on Node.js.
- **blake3_wasm_small** serves Bun users and bundle-size-sensitive contexts (31.8 KB vs 46.8 KB).

## Publishing

Uses [changesets](https://github.com/changesets/changesets) for version management and changelog.
Prerequisite: `npm i -g @changesets/cli`

Both `@fuzdev/blake3_wasm` and `@fuzdev/blake3_wasm_small` are published to npm.

### Version flow

The root `package.json` (`@fuzdev/blake3`, private) is the changesets anchor. Version flows:

1. Publish script runs `changeset version` (wetrun) — bumps `package.json` + generates `CHANGELOG.md`
2. Publish script syncs version to `Cargo.toml` + `jsr.json`
3. Build runs wasm-pack which copies version from `Cargo.toml` into `pkg/web/package.json`
4. `scripts/patch_npm_package.ts` patches the generated `package.json` with npm metadata
5. Publish script verifies `pkg/web/package.json` version matches before publishing

`changeset publish` is NOT used — the root is private and the publishable dir is a build artifact.
`scripts/publish.ts` handles the full pipeline including `changeset version` and `npm publish`.

### npm package structure

The npm package is ESM (`"type": "module"`). `scripts/patch_npm_package.ts` runs after each
web target build and generates files plus patches `package.json`:

1. `stream.js` — generated from `crates/blake3_wasm_core/stream.ts` via targeted type stripping (single source of truth)
2. `index.js` — Node.js entry: auto-init via `readFileSync` + `init_sync`, imports stream.js
3. `browser.js` — Browser/default entry: `init()` for async init, all exports guarded with `_check()` including `Blake3Hasher` (constructor + static methods)
4. `index.d.ts` — types for all exports including stream functions and `init`/`init_sync`
5. `README.md` — npm-specific README (user-facing content only, no build/dev/publish sections, package-specific title)

Package exports include `"./package.json"` self-reference and `"."` with `"node"` / `"default"`
conditions so Node.js gets zero-config sync init and bundlers (Vite etc.) get the `browser.js`
entry which works with `new URL(...)` WASM patterns natively.

### Publish steps

```bash
changeset                          # add a changeset (during development)
deno task publish                  # dry-run: sync, check, build, validate (no version bump)
deno task publish --wetrun         # changeset version + sync + check + build + validate + publish
# git add . && git commit -m "vX.Y.Z" && git tag vX.Y.Z && git push && git push --tags
```

`deno task publish` runs these steps (fail-fast):

1. Preflight: find changeset CLI, check git cleanliness (fail on `--wetrun`, skipped in retry mode),
   check `npm whoami` (wetrun only)
2. Bump version: run `changeset version` (wetrun only), or read current version (dry-run).
   Writes `.changeset/.publish-in-progress` sentinel after bumping; detects retry via sentinel.
   Retry mode: sentinel present → skip version bump, proceed from current `package.json` version
3. Sync version to `Cargo.toml` and `jsr.json` (wetrun only — dry-run reports state)
4. `deno task check` (typecheck + test + clippy + cargo fmt --all + deno fmt)
5. `deno task build:wasm` (all WASM targets)
6. Verify `pkg/web/package.json` version matches
7. `deno task validate:npm` + `deno task validate:size` + correctness vs test vectors (`scripts/compare.ts`) + `deno task validate:compile`
8. `npm publish` from `pkg/web/` dir (`--dry-run` unless `--wetrun`), skipping already-published packages. Remove sentinel after all succeed

### JSR

Deferred — requires inlining WASM as base64 into a JS module (the idiomatic JSR pattern,
used by `@std/crypto` via `@deno/wasmbuild --inline`). Config exists in
`crates/blake3_wasm/jsr.json` (`@fuzdev/blake3`) and `crates/blake3_wasm_small/jsr.json`
(`@fuzdev/blake3-small`). JSR packages are created on jsr.io but not yet published.

### Version source

| Source                                   | Managed by                                     | Purpose                                 |
| ---------------------------------------- | ---------------------------------------------- | --------------------------------------- |
| `package.json` version                   | `scripts/publish.ts` (via `changeset version`) | Changesets anchor, CHANGELOG generation |
| `Cargo.toml [workspace.package] version` | `scripts/publish.ts`                           | Rust workspace, wasm-pack reads this    |
| `jsr.json` versions                      | `scripts/publish.ts`                           | JSR metadata (deferred)                 |
| `pkg/web/package.json` version           | wasm-pack (build)                              | npm publish reads this                  |
